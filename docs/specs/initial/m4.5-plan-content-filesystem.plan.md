# Plan: M4.5 — Plan Content Filesystem Layer

## Context

**M4 (Project Planning)** shipped the full project planning model: project lifecycle (create from spec, status transitions), milestones with status validation, task groups, tasks with dependencies, plan content CRUD via `plan_content` SQLite table, MCP tools for project planning, and a full UI with project pages, task views (dependency graph, swimlane, Eisenhower matrix), plan tab with inline BlockNote editor, and workspace overview with project cards.

**Problem**: Plan content is stored in the `plan_content` SQLite table — a transient execution store. This contradicts the filesystem philosophy (`docs/specs/initial/context/filesystem.md`) where permanent knowledge lives as git-trackable markdown files under `.engy/`. Milestone plans are knowledge artifacts (the "why" and "how" of a milestone) that should survive DB resets, be diffable, and be readable without the app running.

**Goal**: Move plan content to the filesystem as `m{N}-{slug}.plan.md` files inside the spec directory, making them the source of truth. Reduce DB orchestration by removing the `plan_content` table entirely from the schema.

---

## Design Decisions

### 1. Milestones: Keep in DB, milestones are optional

**Decision**: Keep milestones in SQLite with auto-increment IDs. Milestones are optional — projects that don't need milestone breakdown have zero milestone rows in DB and no plan files.

**Justification**:
- When milestones exist, they stay in SQLite for referential integrity (tasks reference via `milestoneId` FK, cascade deletes for task groups and tasks).
- When a project has no milestones, nothing is in DB for milestones — `spec.md` is the implicit plan.
- The plan service only creates files when milestones exist; no empty directories are scaffolded.
- The `m{N}` convention is used in the *filename* only (`m1-setup.plan.md`), derived from `sortOrder + 1`. The DB milestone `id` remains the internal reference for tasks and groups.

### 2. Plan content: Filesystem as source of truth, drop `plan_content` table

**Decision**: Plan content is read from and written to `m{N}-{slug}.plan.md` files directly in the spec directory. The `plan_content` table is removed from the migration entirely — it will not exist in the database.

**Naming convention**: `m{sortOrder + 1}-{milestone-title-slug}.plan.md` (e.g., `m1-setup.plan.md`, `m2-authentication.plan.md`).

**Where files live**:
```
.engy/specs/{spec-slug}/
  spec.md
  context/
  m1-setup.plan.md
  m2-auth.plan.md
  m3-payments.plan.md
```

### 3. No plan/ subdirectory

**Decision**: Plan files live directly in the spec directory root, not in a `plan/` subdirectory.

**Justification**: The spec dir is the single home for all spec artifacts. A subdirectory adds nesting without benefit — plan files are first-class artifacts alongside `spec.md` and `context/`.

### 4. Lazy file creation

The plan files are created lazily — when the first plan file is written for a project, not at project creation. This avoids empty directories cluttering the filesystem.

### 5. Linking project to spec directory

Projects already store `specPath` (the spec directory name, e.g., `1_auth-revamp`). The plan service uses this to resolve the filesystem path: `{workspace-dir}/specs/{specPath}/`.

## Functional Requirements

### Plan File Service (FR 1-6)

1. **Write plan file**: Given a workspace, spec slug, milestone sort order, milestone title, and markdown content, the service shall write a file at `{specs-dir}/{spec-slug}/m{sortOrder+1}-{title-slug}.plan.md`.

2. **Read plan file**: Given a workspace, spec slug, and milestone filename, the service shall return the file contents. Returns `null` if the file does not exist.

3. **List plan files**: Given a workspace and spec slug, the service shall return a sorted list of filenames in the spec directory that match the `.plan.md` extension. Returns empty array if no plan files exist or directory does not exist.

4. **Delete plan file**: Given a workspace, spec slug, and filename, the service shall delete the file. Throws if file does not exist.

5. **Rename plan file**: When a milestone's title or sort order changes, the plan file must be renamed to match the new `m{N}-{slug}.plan.md` convention. If the old file does not exist, this is a no-op.

6. **Path safety**: All file operations shall use `validatePath()` (same pattern as spec service) to prevent directory traversal.

### Router Changes (FR 7-10)

7. **planContent.get**: Shall resolve the milestone's project -> project's specPath -> read plan file from filesystem. Returns `null` if no file exists. No DB fallback (table does not exist).

8. **planContent.upsert**: Shall write plan content to the filesystem only. Returns early with a message if project has no `specPath`. Does NOT write to any DB table.

9. **planContent.delete**: Shall delete the plan file from filesystem only.

10. **planContent.list**: NEW procedure. Given a projectId, returns all plan files with their milestone associations.

### MCP Changes (FR 11-12)

11. **planMilestone MCP tool**: Shall write plan content to filesystem instead of `plan_content` table. Otherwise same interface.

12. **getProjectOverview MCP tool**: No change needed (does not read plan content).

### Milestone Update Hook (FR 13)

13. **Milestone rename/reorder**: When a milestone's title or sortOrder is updated, if the project has a specPath, the plan file shall be renamed to match. This is best-effort (no error if file doesn't exist).

## Migration

### Existing `plan_content` table

- The table is removed from the existing migration — it will not exist in any environment.
- Since this is pre-production with no real data, there is no need for a fallback or grace period.
- Dev DB is dropped and recreated from scratch: `rm .dev-engy/engy.db && pnpm dev`.

### Existing projects without specPath

- Projects without `specPath` (e.g., the default project, or manually created projects) cannot have filesystem-backed plans. The router returns `null` for these, and `upsert` returns early with a clear message.
- This is expected: only spec-derived projects have a spec directory for plan files.

### Potential Issues

1. **Resolution chain complexity**: `planContent.get` and `planContent.upsert` currently take only `milestoneId`. To resolve to a filesystem path, they need: milestone -> project (via `milestones.projectId`) -> project's `specPath` -> workspace (via `projects.workspaceId`) -> workspace slug -> workspace dir. This is 3 DB lookups per plan read/write. **Mitigation**: Acceptable for a single-user tool with SQLite (microsecond queries).

2. **Projects without specPath**: Default projects and manually-created projects have no `specPath`. The plan correctly handles this (returns `null` / returns early with message).

3. **Concurrent filename conflicts**: If two milestones have the same title and sort order (shouldn't happen but possible via direct DB manipulation), their plan files would collide. **Mitigation**: milestones within a project should have unique sortOrder. The filename includes both sort order AND title slug, making collisions very unlikely.

4. **Milestone rename cascading to plan file**: FR 13 requires updating the plan filename when milestone title/sortOrder changes. The milestone router needs to: (a) compute old filename from old values, (b) compute new filename from new values, (c) call `renamePlanFile`. This requires reading the milestone's current state before the update (already done for status validation).

### Phase Ordering

Correct. Phase 0 (migration cleanup) is safe to do first since `plan_content` is not currently used in Phase 1 or 2. Phase 1 (service layer) has no dependencies on Phase 2. Phase 2 depends on Phase 1. All phases are independently `pnpm blt`-green since:
- Phase 0 removes unused schema (knip will catch any lingering references)
- Phase 1 adds new files only (no breakage)
- Phase 2 modifies existing routers but preserves their interfaces

### Verdict

**LGTM** — plan is complete, correctly scoped, and implementable in 3 phases.
